# Multi-profile pipeline: one tick iterates active profiles serially

The pipeline ran a single profile (`config.profile_id`, default `default`); a second tenant's profile would never run. Each scheduled tick now lists active profiles and runs the existing single-profile pipeline for each, **serially**. Serial — not parallel — because inference is Ollama on a single GPU, so concurrent profiles would only contend for the same device; serial execution also makes per-profile `runs`/`integration_events` clean and bounds load predictably. The Owner absorbs all LLM cost (acceptable for a household; there is no per-profile billing). To keep an unbounded profile from becoming unbounded GPU/token spend, a `max_monitors_per_profile` cap (default 25) is enforced at Monitor-create time. Per-profile independent schedules were deferred — a single global cadence looping all profiles is simpler and sufficient until someone needs different cadences.
