# Inference goes through a shared GPU broker, not direct Ollama calls

**Status: accepted — separate repo confirmed. Broker is `ollama-resource-broker` (https://github.com/preston-bernstein/ollama-resource-broker). HTTP-fronting design pending its own grill.**

With multiple profiles serially driving Ollama, plus other home-lab consumers (estate-scraper vision, LibreChat/LightRAG) and the operator's own GPU use (gaming, video), direct `ollama_host` calls from each service contend blindly for one GPU. A **GPU broker** sits in front of Ollama: a single arbiter that queues requests, applies priority, and **yields** (preempts/drains) when the operator is gaming or Plex is transcoding — emitting events so each caller knows when its work ran or was deferred. Fashion Monitor consumes the broker instead of Ollama directly and needs almost no new backpressure logic: its existing `PENDING` score state already replays listings on the next healthy run when the LLM is unreachable, so a "busy/yielding" response is handled by machinery that already exists.

This is **not greenfield**. A prior home-lab design already exists and is deployed (V3): `ollama-resource-broker`, a process-based resource manager that detects gaming/Plex and preempts + queues Ollama *batch jobs*, with a Tier 2/3 roadmap (configurable preemption, SQLite persistence, priorities, quotas, web UI, Prometheus) that aligns with what multi-tenant Fashion Monitor needs. The broker lives in its **own repo** (confirmed — shared home-lab concern, multiple consumers, independent lifecycle; not a Fashion Monitor feature). **Known gap:** the deployed V3 wraps CLI batch jobs and throttles the Ollama cgroup during gaming; it does **not** front Ollama's HTTP API (`:11434`), which is how Fashion Monitor / LightRAG / estate-scraper actually call it. Closing that gap (an HTTP-fronting queue with priority + yield + events) is the subject of the broker's own design grill; until then Fashion Monitor relies on `PENDING` + the cgroup throttle.
