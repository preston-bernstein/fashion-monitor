# Plan: Stealth Sidecar Migration

## Approach
Add a thin TypeScript HTTP client in `packages/core` that speaks the sidecar's fixed `/v1` contract (contexts/pages/navigate/content/screenshot/health), and rewire Depop's fallback and Poshmark's scraper to drive it instead of a local Patchright/Playwright object. The sidecar exposes no `page.evaluate`/click primitive, so the part of this migration that isn't just endpoint-swapping is moving tile extraction out of in-browser DOM scripts (`depopTileExtractScript`/`poshmarkTileExtractScript`, run today via `page.evaluate`) into Node-side HTML parsing with `cheerio` (already a `packages/core` dependency) against the HTML string `GET /v1/pages/{id}/content` returns. Poshmark's persistent profile maps directly onto the sidecar's existing `user_data_dir` stealth option — no new persistence mechanism needed, just passing the same `data/poshmark-profile` path through on context creation. Once both callers are verified against the sidecar, `playwright/browser.ts` and its four now-unused npm packages (`patchright`, `playwright`, `playwright-extra`, `puppeteer-extra-plugin-stealth`) are deleted from `packages/core` in the same change — no driver flag, no parallel path.

## Architecture
```
apps/cli (scrape.ts / run.ts)
      │
      ▼
PoshmarkScraper.search() / scrapeDepopViaPlaywright()
      │  (platform-specific: build search URL, decide poll predicate)
      ▼
packages/core/src/platforms/stealth-sidecar/
  session.ts   — withEphemeralPage() [Depop], getOrCreatePersistentContext()/
                 closePersistentContext() [Poshmark], pollContent()
  client.ts    — createContext/createPage/navigate/getContent/getScreenshot/
                 closePage/closeContext/checkHealth; retry + timeout +
                 error-typing live here, once, not per caller
  errors.ts    — SidecarUnreachableError, SidecarResponseError
      │  fetch() over HTTP, JSON in/out (base64 for screenshot)
      ▼
Stealth sidecar (scraper-commons, unmodified) — 127.0.0.1:${STEALTH_SIDECAR_URL port}
  FastAPI /v1 routes → single ThreadPoolExecutor(max_workers=1) → patchright
      │
      ▼
extract.ts (per platform) — cheerio-based extractXTilesFromHtml(html, baseUrl)
  replaces the old page.evaluate scripts; feeds into existing normalize.ts
```
Both containers (fashion-monitor's `scraper`/`poshmark` services and the sidecar) must share `network_mode: container:gluetun-scraper` so the sidecar's loopback bind is reachable — see Integration points and Risk areas for the cross-repo gap this creates.

## Data model
No data model changes. This is a transport/launch-mechanism change only — no DB schema, no new tables, scrape output shapes (`Listing`) are unchanged and downstream dedupe/prefilter/scoring are untouched (FR13/AC7).

## API / interface contract
**Sidecar's HTTP contract is fixed and out of scope to change** (see scraper-commons `docs/stealth-sidecar/plan.md`); this section documents the fashion-monitor-side TS functions this migration adds and how each maps onto it. One correction found while reading the actual sidecar source (not just its docs): `app.py`'s `create_context` handler takes the stealth options as a **flat** `Body(default_factory=dict)`, not wrapped under an `"options"` key as `schemas.ContextCreateRequest` implies — the client must POST the flat dict (`{"user_data_dir": "...", ...}`), not `{"options": {...}}`.

| `stealth-sidecar` client function | Sidecar call | Notes |
|---|---|---|
| `checkSidecarHealth()` | `GET /v1/health` | Throws `SidecarUnreachableError` on fetch failure, `SidecarResponseError("unhealthy")` when `status !== "healthy"`. Called at the top of `PoshmarkScraper.search()` and `scrapeDepopViaPlaywright()` (FR12/AC4) |
| `createContext(opts?: { userDataDir?: string })` | `POST /v1/contexts` body `{ user_data_dir? }` | Omit `headless`/launch-arg options entirely — the sidecar's stealth module owns Chromium launch args now, not the caller (see Technology choices) |
| `createPage(contextId)` | `POST /v1/contexts/{id}/pages` | 404 → `SidecarResponseError("not_found")` |
| `navigate(pageId, url, timeoutMs?)` | `POST /v1/pages/{id}/navigate` | `url` must be http/https (sidecar 422s otherwise — never hit in practice since scrapers only build https URLs). `timeout_ms` passed must not exceed the sidecar's configured `op_timeout_ms` (default 30000ms) or the call 422s with `invalid_timeout` — the client's `navigate()` should NOT blindly forward the old 60_000ms goto timeout used by the legacy driver; cap the requested timeout at a safe value below the sidecar default (e.g. 25000ms) or omit it to use the sidecar's own default |
| `getContent(pageId)` | `GET /v1/pages/{id}/content` | Returns raw HTML string |
| `getScreenshot(pageId)` | `GET /v1/pages/{id}/screenshot` | Base64 PNG → `Buffer`, used only by `verify-scrapers.ts` posture capture |
| `closePage(pageId)` | `DELETE /v1/pages/{id}` | Best-effort; swallow 404 (already gone) |
| `closeContext(contextId)` | `DELETE /v1/contexts/{id}` | Best-effort; swallow 404 |

Error mapping (FR6): any `fetch()` rejection (ECONNREFUSED/ECONNRESET/abort-timeout) → `SidecarUnreachableError`; any non-2xx response → `SidecarResponseError` carrying the sidecar's own `{error: {type, message}}` envelope (`invalid_option`/`invalid_url`/`invalid_timeout`/`not_found`/`capacity_exceeded`/`operation_timeout`/`internal_error`/`unhealthy`) plus HTTP status. Both extend a common `SidecarError` base so call sites can do `catch (err) { if (err instanceof SidecarError) log.error(LogEvents.PlatformSidecarUnreachable, ...) }`, distinct from `ScrapeError`/selector-not-found failures (satisfies FR6/AC5).

Retry (FR7): exactly **one** retry, and only when the failure is a connect-level error (`ECONNREFUSED`/`ENOTFOUND`/`ECONNRESET` before any response bytes arrive) — never on a timeout or a non-2xx response, since those mean the request may already have executed against the sidecar's single in-flight worker slot and blindly retrying could double-navigate or leak a context. Fixed 200ms delay before the retry. This resolves FR7's "[retry count TBD]".

## Integration points
- `packages/core/src/platforms/stealth-sidecar/client.ts` — new: low-level HTTP wrapper (create/close context/page, navigate, content, screenshot, health), base-URL resolution, retry/timeout logic. Built on the existing `fetchWithTimeout` in `packages/core/src/lib/http.ts`, not a new HTTP library.
- `packages/core/src/platforms/stealth-sidecar/session.ts` — new: `withEphemeralPage()` (create context+page, run callback, always close both in `finally` — Depop), `getOrCreatePersistentContext(profilePath)`/`closePersistentContext(profilePath)` (module-level `Map<profilePath, Promise<string>>`, mirrors `browser.ts`'s old `persistentContexts` cache — Poshmark), plus `closeAllPersistentContexts(): Promise<void>` which closes every context the module's `Map` currently tracks and then clears the Map, mirroring the old zero-arg `closeAllStealthBrowsers()` behavior for callers that don't have a specific `profilePath` on hand, `pollContent(pageId, predicate, {timeoutMs, intervalMs})` (shared poll loop, see Risk areas). The `Map` caches the in-flight `Promise<string>` from `createContext()`, not just the resolved context ID — this closes a check-then-act race where two concurrent calls to `getOrCreatePersistentContext(profilePath)` before the first `createContext()` call resolves would otherwise both see a cache miss and both fire a create call; a second concurrent caller now awaits the same in-flight promise instead. (This is exactly what a new acceptance criterion in requirements.md tests: two concurrent calls result in only one `createContext` HTTP call.) `getOrCreatePersistentContext`/`getOrCreateEphemeralContext` should also reject a `profilePath`/`userDataDir` containing `..` path-traversal segments before passing it to the sidecar, as a cheap defensive check (the value is operator-controlled config today, not external input, so this is a low-priority hardening addition, not a response to a live threat).
- `packages/core/src/platforms/stealth-sidecar/errors.ts` — new: `SidecarError`, `SidecarUnreachableError`, `SidecarResponseError`.
- `packages/core/src/platforms/depop/extract.ts` — `depopTileExtractScript()` (browser-evaluated) replaced by `extractDepopTilesFromHtml(html: string, baseUrl: string): DepopTileRaw[]` using `cheerio`; now calls `parseDepopTileText()` directly instead of duplicating its logic inline (the duplication existed only because `page.evaluate` couldn't serialize a call to a sibling function — that constraint is gone). Two fields specifically need explicit `new URL(value, baseUrl)` resolution: `url` (from the anchor's raw `href` attribute — cheerio's `.attr('href')` does NOT auto-resolve like the browser's live `.href` getter did) and `image` (from `.attr('src')`, which can be `null` — must null-guard before calling `new URL()`, since `new URL(null, baseUrl)` throws).
- `packages/core/src/platforms/depop/playwright-fallback.ts` — `scrapeDepopViaPlaywright()` rewritten: `checkSidecarHealth()` → `withEphemeralPage()` → `navigate()` → poll up to 3x (`pollContent` with a "tiles found" predicate, same 4s/2s/2s cadence as today) → `extractDepopTilesFromHtml()`. Drops the cookie-consent-banner click (no click primitive in the sidecar API); harmless since `content()` returns the full rendered DOM regardless of the banner's accept state. This assumption must be verified with a real fixture-HTML regression test (capture actual Depop response HTML with the banner unaccepted, confirm tile extraction still succeeds) before relying on it — not just asserted from code inspection.
- `packages/core/src/platforms/poshmark/extract.ts` — `poshmarkTileExtractScript()` replaced by `extractPoshmarkTilesFromHtml(html, baseUrl)` using `cheerio`.
- `packages/core/src/platforms/poshmark/scraper.ts` — `getPoshmarkContext()` returns a `contextId: string` (was `BrowserContext`) via `getOrCreatePersistentContext(profilePath)`; `scrapePoshmarkQuery(contextId, query)` rewritten: `createPage()` → `navigate()` → `pollContent()` replacing `page.waitForSelector(...)` → fixed 2s settle sleep (unchanged) → `getContent()` → `extractPoshmarkTilesFromHtml()` → `closePage()`. `closePoshmarkContext()` keeps its existing zero-arg signature (unchanged at both CLI call sites) but now calls the new `closeAllPersistentContexts()` instead of the old `closeAllStealthBrowsers()` — this matters because `apps/cli`'s `forEachProfileSerially` loop can open multiple profiles' persistent contexts in one process run, and `closePoshmarkContext()` has no way to know which specific `profilePath` to pass since it's called with zero arguments from a top-level `finally` block in both `scrape.ts` and `run.ts`.
- `apps/cli/src/scrape.ts`, `apps/cli/src/run.ts` — remove the `closeAllStealthBrowsers` import/call entirely. Depop no longer holds any cross-call browser state (each call owns its own context/page lifecycle via `withEphemeralPage`), so `closePoshmarkContext()` — already called in both files' `finally` blocks — is the only cleanup needed.
- `scripts/verify-scrapers.ts` — remove `DRIVER_MATRIX`, `PLAYWRIGHT_STEALTH_DRIVER` handling, `getEphemeralBrowserDriver`/`resolveStealthDriver` imports, and the whole per-driver matrix loop (`runDriverMatrix`, `runDepopMatrixRow`, `runPoshmarkMatrixRow`'s driver-parameterization) — there is exactly one path now (AC8). `capturePosture()` rewritten to use the new sidecar client directly (create context → page → navigate → screenshot → close) for its diagnostic screenshot capture; Depop/Poshmark each get one report row instead of two.
- `packages/core/tests/platforms/playwright-browser.test.ts` — delete (tests the retired module).
- `packages/core/tests/platforms/stealth-sidecar/client.test.ts` — new: mocked `fetch`, covering retry/timeout/error-typing, including the `invalid_url`/`invalid_timeout` 422 cases and the `navigate()` timeout-capping behavior described above.
- `packages/core/tests/platforms/depop-extract.test.ts` — updated to test new cheerio-based extraction.
- `packages/core/tests/platforms/depop-playwright-fallback.test.ts` — updated to mock sidecar client.
- `packages/core/tests/platforms/poshmark-extract.test.ts` — updated to test new cheerio-based extraction.
- `packages/core/tests/platforms/poshmark.test.ts` — updated to mock sidecar client.
- `packages/core/package.json` — remove `patchright`, `playwright`, `playwright-extra`, `puppeteer-extra-plugin-stealth` from `dependencies` (all four are used only by the retired driver — confirmed via repo-wide grep; the root `@playwright/test` used by `tests/e2e/*.spec.ts` for web-app E2E is a separate package, untouched). Remove the `playwright install chromium` step from the `postinstall` script — no browser binary is launched locally anymore.
- `Dockerfile` — remove `RUN pnpm exec playwright install --with-deps chromium` (and its comment) — shrinks the fashion-monitor image since browser automation now lives entirely in the sidecar's own container.
- `docker-compose.yml` — do NOT add a `stealth-sidecar` service. Only add a `STEALTH_SIDECAR_URL` environment entry to the `scraper` and `poshmark` service definitions' `environment:` block (or rely on it being read from `.env` via `env_file: .env`, matching the existing pattern — check which is more consistent with how other services in this file already pass config and use that). No `depends_on` on any sidecar service, since none is defined in this file. Standing up the sidecar as an actual reachable container in the `gluetun-scraper` network namespace on the desktop is an explicit, tracked, cross-repo follow-up owned by home-infra (ADR 0014/0015, shared-service deploy/attach wiring) and/or scraper-commons (needs to ship a container image) — NOT resolved by this feature. Until that follow-up lands, `scraper`/`poshmark` running on the real desktop deploy will fail-fast with `SidecarUnreachableError` when they try to reach `STEALTH_SIDECAR_URL` (this is expected, correct behavior per FR12 — a clear, fast, visible failure — not a silent fallback, and does not affect any other service in the compose file since nothing else depends on the sidecar). This scope correction matters because `docker compose up -d` (this repo's actual deploy command, run via `make deploy`) builds/pulls ALL services in the compose file — a missing image for a new `stealth-sidecar` service definition could abort the entire deploy, taking down unrelated services (dashboard, ntfy, grafana, mcp-server) that have nothing to do with this migration.
- `.env.example` — replace `PLAYWRIGHT_STEALTH_DRIVER=legacy` with `STEALTH_SIDECAR_URL=http://127.0.0.1:8000` (sidecar's own default `PORT`), satisfying FR8.

## Technology choices
- **`cheerio`** (already a `packages/core` dependency, used by Vestiaire's fetch path) for HTML parsing — reuses an existing library instead of introducing a new HTML/DOM parser, and its jQuery-like `$()` API is close enough to the old `document.querySelectorAll` code that the port is mechanical for most of the selector logic.
- No new HTTP client library — `fetchWithTimeout`/`fetchJson` in `packages/core/src/lib/http.ts` already provide timeout-bounded `fetch()`; the sidecar client builds directly on that rather than adding `axios`/`got`/etc.
- No new browser-automation library in fashion-monitor (constraint from requirements) — Chromium launch args, headless mode, and stealth fingerprinting move entirely into the sidecar's `scraper_commons.stealth` module; fashion-monitor's client no longer needs to know or care about `--no-sandbox`-style launch flags.

## Risk areas
1. **Production desktop reachability is an explicit, accepted gap, not resolved by this feature.** The sidecar has no container image in scraper-commons yet. This feature's own end-to-end verification runs the sidecar as a local process (`python scripts/run_sidecar.py`) against real Depop/Poshmark scrapers on this dev machine — sufficient to verify the code migration is correct and to justify retiring the legacy driver per the user's explicit "no parallel driver" requirement. It is NOT sufficient to verify the real desktop-deployed compose stack, since `scraper`/`poshmark` containers there run inside the `gluetun-scraper` network namespace and a bare host process is not reachable from inside that namespace. Real production scraping will fail-fast (correctly, per FR12) until home-infra or scraper-commons closes this cross-repo gap — tracked in `home-infra/docs/plans/stealth-sidecar-rollout.md`, not invented by this feature.
2. **No `page.evaluate`/click primitive in the sidecar API is the biggest behavioral gap.** Moving extraction from in-browser DOM scripts to Node-side `cheerio` parsing of `content()` is mechanical for selector logic but not risk-free: cheerio's `.attr('href')`/`.attr('src')` return raw (possibly relative) attribute values, unlike the browser's live `HTMLAnchorElement.href`/`.src` getters which auto-resolve to absolute URLs. Every extracted `url`/`image` needs explicit `new URL(value, baseUrl)` resolution — an easy thing to miss, and only verifiable against real fixture HTML, not by code inspection alone.
3. **Polling replaces native Playwright waits, adding real latency under a single-worker-thread sidecar.** Depop's 3x tile-retry loop and Poshmark's `waitForSelector(..., {timeout: 30_000})` both become client-side loops of `navigate`/`content` HTTP calls — each one a full round trip that queues behind whatever else the sidecar's one worker thread is doing (ADR 0005). If the poll interval/budget is copied over unchanged, it may not leave enough headroom for that queuing under real concurrent load (Depop + Poshmark alternating, or a second sidecar consumer per scraper-commons' rollout plan — estate-scraper). This should be measured empirically during live verification (see steps.md) rather than assumed safe — if real round-trip latency under the sidecar's single-worker-thread queueing eats meaningfully into the 4s/2s/2s or 30s budgets, the constants should be adjusted based on that measurement, not shipped unchanged on faith.
4. **Retry policy (FR7) is a judgment call filling an explicit "[TBD]".** One retry, connect-errors only, is a reasoned but unverified choice — it could under-retry a sidecar that's mid-restart (connection refused briefly, then accepting) or, less likely, mask a genuinely down sidecar for one extra round trip before failing fast (AC4).
5. **Client-side context cache can go stale across a sidecar restart.** `getOrCreatePersistentContext`'s `Map<profilePath, contextId>` only lives as long as the Node process; if the sidecar process restarts between scrape runs (its own registry is in-memory, non-persistent by design), a cached `contextId` from a prior run is silently invalid. The client must treat a 404 `not_found` on `createPage`/`navigate` against a cached context as "recreate," not a hard failure — this is a new failure mode the old in-process design never had (a local `BrowserContext` object couldn't outlive the process that created the cache entry). Note: the sidecar itself already has its own idle-context reaper (`idle_timeout_s=300` per scraper-commons' existing implementation) that bounds (does not eliminate, but bounds to ~5 minutes) the blast radius of a context orphaned by a Node-process crash that skips the `finally` cleanup block — this is an existing capability of the already-built sidecar, not something this feature needs to add.

## Security notes
- Never log raw HTML content from `getContent()`/`getScreenshot()` responses or full navigated URLs with query parameters — log only extracted tile counts, platform name, and the error type/message on failure, consistent with existing logging conventions in this codebase.
