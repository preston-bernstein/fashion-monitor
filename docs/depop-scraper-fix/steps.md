# Steps: Fix the Depop Scraper

## Prerequisites
None.

## Implementation steps

### Step 1: Live-verify the real endpoint, block behavior, and DOM hydration BEFORE building anything
**What**: Perform ONE careful, low-volume live-verification pass against Depop before the fixture, normalizer, scraper, or fallback work begins — everything downstream in this document was previously being built on an unconfirmed guess about the real API response shape and Cloudflare behavior, and this step exists to remove that guess first. Confirm: (a) the actual current products-search endpoint — verify `https://www.depop.com/presentation/api/v1/search/products/?what=<query>&limit=24&country=us&currency=USD&from=in_country_search&include_like_count=true` is still live and correctly targeted, adjusting if Depop has changed it since this plan was drafted; (b) whether a plain HTTP call (impit, `browser:"firefox"`, no cookie warm-up, no custom headers) to that endpoint ever succeeds without a Cloudflare block, or is blocked on essentially every attempt; (c) the real success-response JSON shape if any call succeeds, OR the real Cloudflare-block response shape (status code, headers such as `server: cloudflare`, body markers) if every call is blocked; (d) whether the Playwright/patchright fallback's own page-hydration actually reaches real product tiles — launch one real patchright browser session against the search page, wait for hydration, and check whether product tiles render; if they do, capture the real CSS selector for a tile. Before running any check that shells out through `scripts/verify-scrapers.ts`, run `pnpm build` (or the `packages/core` build script) first — that script imports from `packages/core/dist/`, not `src/`, so skipping the build would silently verify stale, pre-investigation compiled code instead of anything real. This is ONE careful, low-volume investigation (a single query such as "corduroy jacket", a small fixed number of attempts) — do not loop or retry this step to chase a cleaner result; a single honest negative result (e.g. "blocked every time") is itself the needed finding. Write down the findings (confirmed endpoint, plain-HTTP success/block rate, real JSON shape or real block shape, real DOM selector or confirmed-blocked hydration) — every later step in this document that builds the fixture, normalizer, ScrapFly-detection heuristic, or DOM-tile selector depends on this step and must use these real findings, not a guess.
**Files**: none (read-only investigation; findings are recorded for later steps to consume, no production files are touched in this step)
**Test**: This step's test is that a written finding exists for all four items: (1) confirmed endpoint URL, (2) plain-HTTP success rate against it (including whether it ever succeeds at all), (3) the real success JSON shape OR the real block-response shape, (4) the real DOM tile CSS selector OR confirmed-blocked/no-hydration status. Later steps' "passes offline" test claims are only honest to the extent they build on these recorded findings rather than an assumption.
**Depends on**: none
**Parallelizable**: No — this step blocks nearly every step below and must run first.

### Step 8a: Rewrite scraper.ts's primary HTTP tier (simple call, no cookie/header machinery)
**What**: Rewrite `packages/core/src/platforms/depop/scraper.ts`'s `searchViaHttp` method for a SIMPLE primary HTTP call: a plain GET to `buildDepopProductsApiUrl(query)` via impit, with NO cookie-harvest warm-up and NO generated `depop-device-id`/`depop-search-id`/`depop-session-id` headers — that machinery is deliberately deferred to the separate, conditional Step 8b, built only if Step 1's findings warrant it. Keep the existing 3-attempt/1500ms+attempt*1000ms backoff loop wrapping ONLY this primary HTTP tier. On Cloudflare-challenge detection (403/429 with a `server: cloudflare` header, or body containing `cdn-cgi`/`Forbidden`/`Just a moment`/`Attention Required`), escalate ONCE to `fetchDepopViaScrapfly` — a one-shot escalation, not independently retried 3x by the primary tier's backoff loop (misreading this would 3x paid ScrapFly spend per query). On any remaining failure, call the Playwright fallback — also a one-shot escalation, never looped. Emit log events for each tier's success/failure per Step 3's event IDs. Preserve the existing `export { parseDepopProducts } from "./normalize.js";` re-export line used by `depop.test.ts` — a full-file rewrite can easily drop it since it isn't part of the class body.
**Files**: `packages/core/src/platforms/depop/scraper.ts`
**Test**: Run `pnpm run test packages/core/tests/platforms/depop-scraper-http.test.ts -- --reporter=verbose` and confirm cascade test cases pass with fixture data (no live calls). This validates cascade wiring and the fixture's provisional shape — it does not establish that Depop's real endpoint actually behaves this way; only Step 1's findings establish that.
**Depends on**: Step 1, Step 3, Step 4, Step 5, Step 7
**Parallelizable**: No

## Rollback plan

- **Step 1**: This is a live-network investigation, not a code change — there is nothing to `git revert` here. The real risk is live-network exposure: tripping Cloudflare/anti-bot flags on the target endpoint, and, if a ScrapFly call happens to be exercised as part of the check, spending shared paid ScrapFly quota. The mitigation is procedural, not code-level: run it ONCE, carefully, at low volume (a single query, a small fixed number of attempts) — do not loop or retry this step to chase a cleaner result. If it trips a block, that block result IS the finding; record it and move on rather than re-running immediately.
- **Steps 2–7**: new files or isolated additions (fixture, log events + doc sync, parse-rsc URL builder, fetch-scrapfly.ts + live-env.ts flag, extract.ts, normalize.ts); `git checkout` reverts deletions/additions.
- **Steps 8a–8b**: substantial rewrite to `scraper.ts`'s `searchViaHttp`; `git checkout` reverts to the prior (dead but syntactically intact) implementation. Step 8b may not exist at all if Step 1's findings justified skipping it — in that case there is nothing to roll back.
- **Step 9**: substantial rewrite to `playwright-fallback.ts`; `git checkout` reverts.
- **Steps 10–12**: test/doc updates; `git checkout` reverts.

If the new scraper fails after shipping (e.g. Depop changes its API again), `git reset --hard origin/main` restores the prior working branch state. The old code will still be broken, but no new regression is introduced.
