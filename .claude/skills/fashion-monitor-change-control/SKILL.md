# Fashion Monitor — Change Control

How changes are classified, gated, and reviewed in this repo. This skill is the
canon for gating; sibling skills defer to it. Audience: an engineer or agent
with zero prior context. All volatile facts date-stamped 2026-07-02.

Vocabulary: use `CONTEXT.md` terms exactly (Monitor, Taste, Profile, Score,
Connection, Pipeline, Feedback, ...). Definitions live there — read it first.

## Where decisions are recorded (two ADR series — do not confuse them)

| Series | Location | Style | Examples |
|---|---|---|---|
| Legacy, 3-digit | `spec/06-decisions.md` (one file, ADR-001..011) | Decision / Context / Why / Why-not sections | ADR-006 Vinted, ADR-007 Taste-in-DB, ADR-010 public repo, ADR-011 alerts |
| Current, 4-digit | `docs/adr/0001..0006-*.md` (one file each) | Short prose: title, bold status line, one dense paragraph with decision + context + why + rejected alternatives | ADR-0004 dormant login Connections, ADR-0005 multi-profile, ADR-0006 GPU broker |

As of 2026-07-02, `docs/adr/0003..0006` and `docs/plans/self-service-onboarding.md`
are **untracked** (in-flight, not yet committed). Treat them as binding anyway —
they are the newest decisions.

How decisions flow (see `docs/plans/self-service-onboarding.md` header for the
worked example): a **grilling session** (adversarial design interview, often via
the `/grill-with-docs` skill) produces a plan doc in `docs/plans/`; decisions
crystallise into ADRs in `docs/adr/`; new or changed vocabulary lands in
`CONTEXT.md` in the same change. All three move together.

## Non-negotiables (each: rule → rationale → evidence)

### 1. No AI/agent attribution — ABSOLUTE

Rule: never add `Co-authored-by`, `Signed-off-by`, "Generated by AI/Cursor/
Copilot" or any agent credit to commits, PR descriptions, code comments, or
docs. Never set git `user.name`/`user.email` to an agent identity.
Rationale: the human owner owns all authorship.
Evidence: `.cursor/rules/no-agent-attribution.mdc` (`alwaysApply: true`).
No exception exists; this overrides any tool default that appends attribution.

### 2. Vocabulary canon — CONTEXT.md wins

Rule: every change (code identifiers where practical, docs, UI copy, commit
messages) uses `CONTEXT.md` canonical terms and respects its explicit
"_Avoid_" lists (e.g. "Monitor", never "Saved Search"/"Search Group" in prose).
If a change shifts meaning or adds a concept, update `CONTEXT.md` in the same
change, as part of the ADR/plan flow above.
Rationale: the DB schema already drifted from the vocabulary once — migrations
`012_search_groups.sql`/`013_unify_search_groups.sql` renamed/unified the
Monitor concept into the `search_groups` table (the repo's one schema
reversal), while the canonical user-facing term stayed **Monitor**. Without an
enforced canon, code, docs, and conversation diverge.
Evidence: `CONTEXT.md` (currently itself modified in the working tree —
Invite/User/Interface-hierarchy updates from the onboarding grill);
`packages/core/src/storage/migrations/012_search_groups.sql`, `013_unify_search_groups.sql`.

### 3. pnpm only — never npm/yarn

Rule: all install/build/test commands use `pnpm` (workspace, `packageManager`
pnpm@9.15.0, Node >= 24 per `engines` in root `package.json`). Never run
`npm install`/`npm ci` here; never commit a `package-lock.json`.
Rationale/incident: an npm lockfile and build artifacts WERE committed and had
to be purged — commit `14c6b26` removed `apps/web/package-lock.json`
(5,855 lines) plus tracked `tsconfig.tsbuildinfo` files; commit `8cd39fb` then
added `**/package-lock.json` to `.gitignore`. Verify with:

```bash
git log --oneline | grep -E '8cd39fb|14c6b26'
git show --stat 14c6b26 | head
```

Known consequence still live as of 2026-07-02: `.github/workflows/ci.yml` and
`live-smoke.yml` still use `node-version: "20"`, `cache: npm`, `npm ci` — CI is
broken as written (npm cannot resolve a pnpm workspace with no lockfile), and
`docs/SMOKE.md` echoes the npm commands. Fixing that is a legitimate class-(b)
code change; do not "fix" it by committing npm lockfiles.

### 4. Public-repo hygiene (ADR-010, spec/06-decisions.md)

Rule: these never enter the repo: `.env` (the encryption root secret lives
there), personal `config.yaml`, `data/` (SQLite DB), the Poshmark browser
profile (`data/poshmark-profile/`), anything matching `*credentials*`/`*.pem`.
`config.example.yaml` and `.env.example` are the committed stand-ins.
Rationale: the repo is public by decision (portfolio value); one leaked secret
or personal Taste config is permanent.
Evidence: ADR-010 in `spec/06-decisions.md`; `.gitignore` (verify it still
covers `.env`, `config.yaml`, `data/`, `poshmark-profile/`, `**/package-lock.json`).
Assumption (A2 — assumed, confirm with owner): also keep personal measurements
and home-network hostnames/IPs out of committed docs and skills.

### 5. Fenced/dormant features — each has an explicit exit gate

Do NOT enable, remove, or "clean up" these without satisfying their gate:

| Fenced thing | Current state | Gate to change it | Source of the fence |
|---|---|---|---|
| Login-based platform Connections (Poshmark/Depop/Vestiaire sessions) | Ship DORMANT: off by default, per-platform risk acknowledgment required | (1) per-platform ToS research producing honest risk copy AND (2) measured anonymous-vs-logged-in result lift | `docs/adr/0004-tiered-connections-login-dormant.md`; Spikes section of `docs/plans/self-service-onboarding.md` |
| `playwright-extra` + `puppeteer-extra-plugin-stealth` | In use (`packages/core/src/platforms/playwright/browser.ts`) | Patchright passes live smoke on Depop/Poshmark | `docs/playwright-stealth-pilot.md` § "Do not remove yet" |
| Vinted platform | Disabled (`vinted: false` in `config.example.yaml`; no `packages/core/src/platforms/vinted/` implementation) | v1 stable + other 5 platforms working; then its own enablement work (Datadome anti-bot) | ADR-006 in `spec/06-decisions.md` |

### 6. Migrations are append-only AND re-run every boot

Rule: schema changes = a NEW numbered `.sql` file in
`packages/core/src/storage/migrations/` (next number after `014_listing_images.sql`).
Never edit an already-shipped migration file.
Repo-specific sharp edge: `migrate()` in `packages/core/src/storage/db.ts` has
**no applied-migrations tracking table** — it sorts and re-executes EVERY .sql
file on EVERY startup. So (a) every migration must be idempotent
(`CREATE TABLE IF NOT EXISTS`, `INSERT OR IGNORE`, column-exists guards — see
the 002/012 special-casing in `db.ts`), and (b) editing an old file is not
"history rewriting", it is a live behavior change on the next boot of every
deployment. Both reasons independently forbid edits.
Also: `docs/plans/stack-modernization.md` § 12 — do not introduce an ORM or a
migration framework; preserve the `migrations/`-next-to-`db.js` layout.

### 7. "Explicitly do NOT change" list (stack-modernization)

`docs/plans/stack-modernization.md` § 12 fences, verbatim topics: `node:sqlite`
adoption (keep `better-sqlite3`), removing `.js` import extensions (NodeNext
ESM), `moduleResolution: bundler` on the backend, frontend major rewrites,
the Fastify auth/session/RBAC design (no Redis sessions), replacing ESLint
with Biome wholesale, ORM/migration frameworks, deployment topology
(Caddy + docker-compose). Changing any of these requires a new ADR that
explicitly supersedes that section.

### 8. Never commit or push without owner instruction

Rule (assumption A2 — assumed, confirm with owner): agents do not run
`git add`/`commit`/`push`/`checkout`/`stash` here unless the owner explicitly
asks in the current session. Rationale: the working tree is routinely used to
stage in-flight design work (see next section); a helpful auto-commit would
publish half-finished migrations to a public repo under rule 4.

### 9. In-flight working tree — hands off (as of 2026-07-02)

The telegram→ntfy alert migration is **uncommitted and internally
inconsistent**: `packages/core/src/alerts/telegram.ts` deleted;
`packages/core/src/alerts/ntfy.ts` untracked; `orchestrator.ts` imports
`createNtfyAlerter`; `config.example.yaml` alert block is ntfy — BUT ADR-011
(`spec/06-decisions.md`) still says "ntfy.sh is not used", `CONTEXT.md` still
says alerts via Telegram, the `Makefile` sync target still echoes `TELEGRAM_*`
env hints, and `apps/cli/src/feedback-bot.ts` is a stub whose comment promises
dashboard feedback that does not exist (`grep -rni feedback packages/api/src`
→ zero hits). The Feedback learning loop has no ingestion path at all.

Rule: do not partially revert, partially commit, or "tidy" someone else's
in-flight migration. Coordinate with the owner. The sanctioned way to finish

the doc reconciliation (ADR-011 supersession, CONTEXT.md, Makefile).

## Change classes and their gates

**(a) Config / Taste tuning.** After first boot the DB is authoritative
(ADR-007, `spec/06-decisions.md`): Taste and system settings live in
`profile_settings`, with every change snapshotted in `config_revisions`.
`config.yaml` is bootstrap-only. Therefore **editing `config.yaml` alone does
NOT change a running system** — tune via the web UI (Curator role) or the MCP
server, which write the DB. See `packages/core/src/core/profile-config.ts`
(`TASTE_SETTING_KEYS`, `SYSTEM_SETTING_KEYS`) and fashion-monitor-config-and-flags
for which axis lives where. Gate: none beyond RBAC, but the change is audited
(`audit_log` + `config_revisions`).

**(b) Code change.** Tests required — unit (vitest) alongside the change;
evidence standards, fixtures, and the live-test policy are defined in
fashion-monitor-validation-and-qa (prefer fixtures; minimize live scrapes —
assumption A2). Must not violate any fence above.

**(c) Schema change.** New numbered idempotent migration per rule 6, plus
updates to the affected repo classes in `packages/core/src/storage/repos/`
and Zod DTOs in `packages/shared` where exposed.

**(d) Design/behavior change.** ADR required in `docs/adr/` (4-digit series,
house style per the table above — short prose, decision + context + why +
rejected alternatives; see fashion-monitor-docs-and-positioning for style
detail). Non-trivial designs are preceded by a grilling-session plan doc in
`docs/plans/`. Vocabulary changes ride along in `CONTEXT.md`. A new ADR that
reverses an old one must name what it supersedes (ADR-011 currently lacks
this and is the live counterexample).

## Decision table: you want to change X

| You want to... | Class | Required gate | Evidence required |
|---|---|---|---|
| Tune Taste (aesthetic prompt, hard-no, price ceiling) | a | Curator role; web UI or MCP (not config.yaml) | `config_revisions` row appears; next Pipeline run uses it |
| Change platform/LLM/alert options on a running system | a | Operator role; DB-backed settings path | `config_revisions` + `audit_log` rows |
| Fix a bug, refactor, add an endpoint | b | Unit tests pass (`pnpm test`); fences respected | Tests covering the change (see validation-and-qa) |
| Fix the broken CI workflows (npm→pnpm, node 20→24) | b | Owner instruction to commit; no npm lockfiles | CI green on a branch; `pnpm` used throughout |
| Add/alter a table, column, or view | c | New numbered idempotent migration; never edit old ones | Migration re-runs cleanly on an already-migrated DB |
| Rename a domain concept | c+d | ADR + CONTEXT.md update + migration if schema follows | 012/013 precedent: term and table may diverge — document it |
| Change alerting, scoring, platform access, or any user-visible behavior | d | ADR (4-digit), usually a `docs/plans/` grill doc first | ADR names rejected alternatives + what it supersedes |
| Enable login Connections | d + fence 5 | ToS research + measured anonymous-vs-logged-in lift | Per-platform risk copy + measurement numbers (see research-methodology) |
| Remove playwright-extra + stealth | b + fence 5 | Rebrowser pilot passes live smoke on Depop/Poshmark | `docs/playwright-stealth-pilot.md` gate satisfied, live smoke output |
| Enable Vinted | d + fence 5 | ADR-006 re-visit once v1 stable | New ADR + platform implementation + tests |
| Touch the ntfy/feedback working tree | — | Owner coordination; run the alerting-feedback campaign | Campaign skill checklist |
| Commit / push anything | — | Explicit owner instruction (A2, assumed) | Owner said so in this session; no attribution (rule 1) |
| Adopt node:sqlite, ORM, Biome-wholesale, bundler resolution, Redis sessions | d | New ADR superseding stack-modernization § 12 | ADR + migration plan |

## When NOT to use this skill

- Running or deploying the system → fashion-monitor-run-and-operate.
- Setting up a dev environment → fashion-monitor-build-and-env.
- Test taxonomy, fixtures, evidence standards for a change you already know
  is allowed → fashion-monitor-validation-and-qa.
- ADR/doc prose style and public-repo positioning detail →
  fashion-monitor-docs-and-positioning.
- Executing the two sanctioned campaigns →
  fashion-monitor-alerting-feedback-campaign (ntfy + Feedback loop),
  fashion-monitor-multi-profile-campaign (ADR-0005 Phase 1).
- History of why things are the way they are →
  fashion-monitor-failure-archaeology.
